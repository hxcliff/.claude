# FFI 专项审计项

## L1 未使用的绑定

- 声明了 FFI 函数签名但无任何调用方
- 生成的绑定文件中导出了目标库未实际提供的符号
- 辅助转换函数（marshal / unmarshal）定义了但未使用

## L2 冗余转换层

- 同一数据在 FFI 边界被反复编解码（如 String -> CString -> *const c_char -> CString -> String）
- 多层 wrapper 逐层透传，无附加逻辑

## L3 绑定生成与同步

- 手写绑定与目标库头文件 / IDL 版本不一致
- 自动生成的绑定文件被手动修改（应修改生成配置而非产物）
- 目标库版本升级后绑定未重新生成

## L5 内存安全

- **所有权不明确**：跨边界传递的指针未明确谁分配、谁释放
- **悬垂指针**：释放后仍持有对侧指针引用
- **双重释放**：两侧各自释放同一块内存
- **缓冲区溢出**：传递数组 / 字符串时未同步传递长度，或长度校验缺失
- **空指针**：未校验从对侧接收的指针是否为 null
- **对齐问题**：跨边界传递的结构体未保证内存对齐一致

## L5 类型编排

- 跨边界的整数类型位宽不匹配（如 Rust `usize` vs C `int`）
- 枚举在两侧的值域不一致或缺少未知值兜底
- 结构体布局未显式指定（缺少 `#[repr(C)]` / `@Structure.FieldOrder` 等）
- 布尔值表示不一致（C 的 int vs 语言原生 bool）
- 回调函数签名两侧不匹配

## L5 异常与 panic

- Rust 侧 panic 跨越 FFI 边界（未定义行为，必须在边界处 catch_unwind）
- C++ 异常跨越 FFI 边界未被捕获
- 错误码在边界传递后未检查或被忽略
- 对侧返回的错误信息未正确解码（编码/生命周期问题）

## L5 线程安全

- 非线程安全的对象被跨线程共享到对侧
- 回调函数在非预期线程被调用（如 C 库在内部线程回调，但回调体内操作了主线程资源）
- 全局状态在 FFI 边界两侧被并发访问但无同步机制

## L5 资源泄漏

- 跨边界获取的资源（文件描述符 / 句柄 / 连接）无对应的释放调用
- 异常路径（错误返回 / panic）中未释放已分配的 FFI 资源
- 回调注册后无取消注册机制，导致引用泄漏

## L6 ABI 稳定性

- 动态链接时未锁定目标库的 soname / dylib 版本
- 符号可见性未限制（导出了不应暴露的内部符号）
- 跨平台构建时未处理调用约定差异（cdecl / stdcall / system）
