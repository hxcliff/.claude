# Library 专项审计项

## L1 公共 API 表面积

- 导出了但无任何外部消费者的公共类型/函数/常量
- `pub` / `export` 范围过大：仅内部使用的符号不应公开
- 公共 API 中暴露了内部实现类型（应通过 newtype / 接口隔离）
- 废弃的公共 API 未标注 `@deprecated` / `#[deprecated]`

## L1 Feature / 可选依赖

- feature 之间存在隐式耦合（启用 A 必须同时启用 B 但未声明）
- 默认 feature 集合过大，引入了多数用户不需要的依赖

## L2 不必要的运行时绑定

- 库代码直接依赖具体日志实现（应仅依赖日志门面 / trait）
- 库代码直接依赖具体异步运行时（应通过 trait 或 feature 抽象）
- 库代码直接读取环境变量 / 配置文件（应由调用方注入）
- 库内部启动线程 / 后台任务但未提供关闭机制

## L2 API 人体工学

- 公共函数参数超过 4 个且未使用 builder / options 结构体
- 要求调用方执行多步初始化才能使用核心功能（应提供便捷构造）
- 错误类型过于笼统（单一 Error 枚举包含几十个变体）或过于碎片化

## L3 依赖最小化

- 引入了重量级依赖但仅使用其中一小部分功能
- 传递依赖将不兼容版本引入消费者的依赖图
- 核心功能的依赖应尽量少，可选功能通过 feature 门控

## L4 版本兼容

- 公共 API 签名变更未体现在版本号中（semver 违规）
- 序列化格式变更未提供迁移路径
- 最低支持的语言 / 编译器版本未声明或与实际使用的特性不符

## L5 安全边界

- 库接受用户可控输入但未做校验（将安全责任推给调用方且未文档说明）
- 公共 API 中暴露 `unsafe` 函数但未文档说明安全不变量
- 解析 / 反序列化功能未设置递归深度和输入大小上限
